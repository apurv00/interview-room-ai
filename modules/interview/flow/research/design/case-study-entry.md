# Design / UX — Case Study Interview — Entry Level (0-2 years)

## Topic Sequence (typical order)
1. **Redesign a Feature** — "Redesign the settings page of a restaurant recommendation app. Users complain about too many notifications and hard-to-find options." Classic entry-level whiteboard prompt — scope is narrow (one screen/flow), problem is concrete, and user pain is pre-defined.
2. **Design for a Specific User** — "Design a grocery shopping app for elderly users." Tests user empathy and accessibility thinking. Must ask clarifying questions about the user (tech comfort, physical limitations, goals) before sketching.
3. **Improve an Existing App** — "Pick an app you use daily. What's one thing you'd redesign and why?" Tests critical thinking, design vocabulary, and ability to identify problems in shipped products. Often used as a warm-up or take-home prompt.
4. **Whiteboard a Simple Flow** — "Design the checkout flow for a food delivery app." Tests ability to think in flows (not just screens), consider edge cases (payment failure, address validation, empty cart), and communicate design thinking in real time.
5. **Accessible Redesign** — "Redesign an ATM to be accessible to all users. Show at least one screen and describe alternative interaction methods." Tests inclusive design thinking and ability to consider diverse abilities (vision, motor, cognitive).
6. **Mobile-First Design** — "Design a mobile app for booking study rooms at a university library." Tests understanding of mobile constraints, thumb-friendly zones, progressive disclosure, and real-world context of use.

## Phase Structure
| Phase | Duration | Description |
|-------|----------|-------------|
| Clarify & Ask Questions | 3-5 min | Ask about users, context, constraints, and success criteria. Jumping straight to screens is a red flag. |
| User & Problem Definition | 3-5 min | Define the primary user, their goal, pain points, and key scenarios. Sketch a simple persona or journey snippet. |
| Ideation & Exploration | 5-8 min | Sketch 2-3 rough approaches. Explain tradeoffs between them. Choose one direction and justify why. |
| Wireframe Key Screens | 8-12 min | Sketch 3-5 key screens/states in the flow. Annotate with interaction notes. Narrate decisions aloud. |
| Present & Defend | 3-5 min | Walk through the solution, connecting each decision to user needs. Be ready for interviewer pushback. |

## What Makes This Level Unique
- **Process over polish — explicitly.** Interviewers grade on thinking structure, not wireframe aesthetics. Messy-but-thoughtful sketches beat clean-but-unjustified screens. Narrating your thinking aloud is as important as what you draw.
- **Asking questions is the first test.** Candidates who immediately start sketching fail the exercise. The first 3-5 minutes of clarifying questions (Who is the user? What's the primary goal? What are the constraints?) demonstrate design maturity even at entry level.
- **Scope is intentionally narrow.** Entry-level exercises focus on a single feature or flow, not an entire product. The constraint tests depth of thinking within a small scope: edge cases, error states, empty states, loading states.
- **User empathy must be explicit.** Every design decision should reference the user: "I chose a bottom tab bar because our user is one-handing the phone while walking" — not "bottom tabs are best practice."
- **Exploring alternatives differentiates.** Strong candidates sketch 2-3 approaches and explain tradeoffs before committing. Weak candidates sketch one idea and polish it. Divergent thinking is the signal.
- **Expect curveballs.** Interviewers will add constraints mid-exercise: "Now the user is in a poor network area" or "What if they're visually impaired?" Adaptability and composure under pressure are tested.

## Common Exercises
- Redesign a specific feature of a well-known app (notifications, settings, onboarding)
- Design a simple app for a specific audience (students, parents, elderly, commuters)
- Improve the checkout/booking/registration flow of a service
- Design an accessible version of a physical interaction (ATM, vending machine, kiosk)
- App critique: evaluate a real app and propose specific improvements

## Anti-Patterns
- Jumping to UI without asking who the user is
- Designing one solution without exploring alternatives
- Focusing on visual polish over interaction logic
- Ignoring edge cases (error states, empty states, loading)
- Not considering accessibility at all
- Defending a design stubbornly when interviewer pushes back
- Using only jargon without connecting decisions to user needs
- Treating the exercise as a solo performance — not engaging the interviewer as a collaborator

## Probe Patterns
- User grounding: "Who is this designed for? What's their primary goal right now?"
- Alternative thinking: "You went with option A — what was option B? Why did you reject it?"
- Edge case awareness: "What happens if the user has no items? What does the error state look like?"
- Accessibility check: "How would a user with limited vision interact with this?"
- Constraint adaptation: "The user just told you they hate modals. How does your design change?"
- Prioritization: "You have 2 weeks of engineering time. Which part of this design ships first?"
- Business awareness: "How does this feature affect the company's core metric (engagement, conversion, retention)?"
