# Frontend Engineer — Behavioral Interview — Mid Level (3-6 years)

## Topic Sequence (typical order)
1. **Career Direction** — Why frontend over fullstack/backend; how perspective evolved with experience
2. **Advocating for Accessibility** — Pushing for WCAG compliance when stakeholders wanted to skip it; making the business case
3. **Design-Engineering Tradeoffs** — Pushing back on a design that hurt performance (heavy animations, complex layouts); proposing alternatives preserving design intent
4. **Production Browser Compatibility Crisis** — P0 CSS/JS bug only in Safari/mobile; incident response for visual regression in production
5. **Component Library / Design System Decisions** — Building custom vs. adopting existing; convincing team; migrating styling approaches
6. **Cross-Functional Collaboration** — Working with PMs and designers on feature spec; bridging design mockups and technical constraints
7. **Performance Optimization Story** — Identifying and fixing Core Web Vitals regression; optimizing LCP/CLS; measurable before/after metrics
8. **Technical Debt in UI Codebase** — Balancing cleanup of legacy CSS/components with feature delivery; advocating for refactoring time
9. **Mentoring a Junior Developer** — Helping new team member with frontend skills; code review approach for UI code

## Phase Structure
- **Context setting (5 min):** "How do you stay current with the frontend ecosystem?" — assesses professional identity
- **Deep behavioral dives (25-30 min):** 3-4 stories with extensive follow-ups; cross-functional dynamics and measurable outcomes
- **Reflective close (5 min):** "What's a frontend decision you'd make differently knowing what you know now?"

## What Makes This Level Unique
- Stories must involve **real products with real users** — not side projects
- Expect **cross-functional conflict**: designer wanted X, PM wanted Y, engineering constraints dictated Z
- Must articulate **measurable impact** (performance metrics, user engagement, accessibility audit scores)
- **Ownership** emerges — should have made component/library/architecture decisions, not just implemented them
- Unique to frontend: **design fidelity vs. technical feasibility** tradeoffs ("animation tanked performance below 10fps on mid-range devices")
- Testing for **systematic debugging** of visual/layout issues

## Anti-Patterns (do NOT ask at this level)
- Entry-level "learning a new framework" questions
- Org-wide strategy or multi-team platform decisions (senior territory)
- Generic teamwork/conflict without frontend context
- Accepting vague "we optimized performance" without probing for metrics, tools, before/after

## Probe Patterns
- Performance fix: "What specific metric before and after? How measured? What tool?" (separates practitioners from blog post readers)
- Design conflict: "What alternative did you propose? How did you communicate the constraint to the designer?"
- Accessibility: "What was the business case? Did you reference WCAG levels? How did you prioritize violations?"
- Component library: "Why that library over alternatives? What was the adoption strategy?"
