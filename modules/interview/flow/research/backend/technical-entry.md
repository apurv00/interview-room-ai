# Backend Engineer — Technical Interview — Entry Level (0-2 years)

## Topic Sequence (typical order)
1. **Programming language fundamentals** — Syntax, type systems, memory management basics in primary language (Python, Java, Go, etc.)
2. **Data structures** — Arrays, strings, hash tables, linked lists, stacks, queues, trees (binary trees, BSTs)
3. **Algorithms** — Sorting (merge sort, quick sort), searching (binary search), basic recursion, string manipulation
4. **REST API basics** — HTTP methods (GET/POST/PUT/DELETE), status codes, statelessness, resource-based URIs, request/response structure
5. **SQL and relational databases** — Basic CRUD queries, JOINs, WHERE clauses, GROUP BY, simple indexing concepts, schema basics
6. **Version control (Git)** — Branching, merging, commits, pull requests, basic conflict resolution
7. **Testing fundamentals** — Unit tests, what functional tests verify, test-driven development awareness
8. **Basic web concepts** — Client-server architecture, how HTTP works, DNS basics, what happens when you type a URL
9. **Debugging approach** — How to read error messages, logging strategies, basic profiling awareness
10. **Simple coding problems** — 1-2 LeetCode easy/medium problems focused on arrays, strings, hash maps, or basic tree traversal

## Phase Structure
- **Warm-up (5-10 min):** Technical background chat — "What languages do you know?", "Walk me through a project you've built." No tricks, just calibration.
- **Coding round (25-35 min):** 1-2 algorithm problems at easy-to-medium difficulty. Focus on arrays, strings, hash tables, basic trees.
- **Knowledge Q&A (10-15 min):** Short-answer on REST APIs, SQL basics, HTTP, Git workflow. Tests foundational understanding, not depth.
- **Closing (5 min):** Questions, technology preferences, learning interests.

## What Makes This Level Unique
- **Most companies skip system design entirely** for entry-level backend positions. Some FAANG companies may include a simplified version (basic API design, not distributed systems), but this is the exception.
- Coding problems are **easy-to-medium LeetCode difficulty**. Not hard problems, not trick questions.
- **Academic/bootcamp projects are acceptable context.** Interviewers don't expect production experience.
- **Enthusiasm and curiosity weigh heavily** — interviewers assess potential and learning ability.
- **Breadth over depth** — questions span many topics shallowly rather than deep-diving.
- **Take-home challenges** (60-90 min build a small API) are a common alternative to live coding.
- **AI/LLM coding tools policy** is evolving — some companies now allow or expect candidates to use AI assistants; others prohibit them. Ask about the policy beforehand.

## Anti-Patterns (do NOT ask at this level)
- Full system design (design Twitter, URL shortener with sharding, etc.)
- Distributed systems concepts (consensus, sharding, CAP theorem)
- Advanced concurrency (locks, semaphores, race conditions, deadlocks)
- Database internals (query optimization, execution plans, B-tree internals)
- Microservices architecture and service mesh
- Production-scale performance optimization
- LeetCode hard problems or advanced dynamic programming
- Cloud infrastructure (Kubernetes, Terraform, CI/CD pipeline design)

## Probe Patterns
- **Deep follow-up:** Coding problems (optimize, edge cases, time/space complexity), project walkthrough (what design decisions and why)
- **Moderate depth:** SQL (write a JOIN, then explain what an index does), REST API (design a simple CRUD endpoint, discuss status codes)
- **Surface treatment:** Git (basic familiarity), testing (awareness-level), debugging (approach, not advanced tooling)
